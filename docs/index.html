❄️ Overview

frost lets you manage Snowflake schemas as code — write SQL files, and frost handles the rest.

📂

Declarative SQL

Define tables, views, schemas, procedures, and grants as plain SQL files using CREATE OR ALTER.

🔗

Dependency Graph

frost parses cross-object references and deploys in the correct topological order — automatically.

📊

Data Loading

Load CSV files into Snowflake with checksum-based change tracking. Only changed files are re-loaded.

✅

Data Quality Tests

Define tests in YAML sidecars next to your CSV files. Six built-in test types with auto-discovery.

🤖

Cortex AI Suggestions

When deployments fail, frost can call Snowflake Cortex AI to suggest fixes for your SQL errors.

🔀

Object Lineage

Auto-detect data flow for procedures, functions, tasks, and streams. Full CDC pipeline visibility with zero config.

🔒

Key-Pair Auth

Secure RSA key-pair authentication to Snowflake. No passwords in your config files.

🚀 Quick Start

Get a project running in three commands.

1. Scaffold a project

frost init my-project
cd my-project

This creates:

my-project/
├── frost-config.yml          # Configuration
├── .env.example              # Environment template
├── .gitignore
├── objects/
│   ├── tables/
│   │   └── sample_table.sql  # Example table
│   ├── views/
│   │   └── vw_active_samples.sql
│   ├── schemas/
│   ├── procedures/
│   │   └── refresh_active_samples.sql
│   └── grants/
├── data/
│   ├── sample_users.csv      # Example CSV
│   └── sample_users.yml      # YAML sidecar (types + tests)
└── keys/

2. Configure credentials

cp .env.example .env
# Edit .env with your Snowflake credentials

3. Preview & deploy

# See what frost will do (no changes made)
frost plan

# Deploy to Snowflake
frost deploy

# Load CSV data
frost load

# Run data quality tests
frost test
Tip Use frost deploy --dry-run to preview deployments without executing SQL against Snowflake.

🔤 Variables

frost supports template variables that are substituted in SQL files before parsing and deployment. Variables let you write environment-agnostic SQL and switch values between dev, staging, and production.

How It Works

Use {{variable_name}} placeholders anywhere in your SQL files. frost replaces them with the corresponding value before executing:

-- objects/grants/read_access.sql
GRANT USAGE ON DATABASE {{database}} TO ROLE {{read_role}};
GRANT SELECT ON ALL TABLES IN SCHEMA {{database}}.PUBLIC TO ROLE {{read_role}};

If database is set to PROD_DB and read_role to ANALYST, frost resolves this to:

GRANT USAGE ON DATABASE PROD_DB TO ROLE ANALYST;
GRANT SELECT ON ALL TABLES IN SCHEMA PROD_DB.PUBLIC TO ROLE ANALYST;

Defining Variables

Variables can be set from three sources. Higher-priority sources override lower ones:

PrioritySourceExample
1 (highest)--vars CLI flagfrost deploy --vars '{"database": "PROD_DB"}'
2FROST_VARS env varexport FROST_VARS='{"database": "STAGING_DB"}'
3 (lowest)frost-config.ymlvariables: section in the config file

In frost-config.yml

# frost-config.yml
variables:
  database: DEV_DB
  admin_role: SYSADMIN
  read_role: READ_ONLY_ROLE
  warehouse: COMPUTE_WH

Via Environment Variable

# JSON string — merges with (and overrides) config file values
export FROST_VARS='{"database": "STAGING_DB", "read_role": "ANALYST"}'

Via CLI Flag

# JSON string — highest priority, overrides everything
frost deploy --vars '{"database": "PROD_DB"}'

VS Code Extension

When using the Frost VS Code extension, variables are read automatically from your frost-config.yml. Every command triggered from the sidebar (Plan, Deploy, Deploy Force, Deploy Target, Lineage, Load Data) passes the config file to frost, so all variables defined there are available.

To use different variable values from VS Code:

  • Edit frost-config.yml — change the variables: section directly. The extension picks up changes on the next command run.
  • Set FROST_VARS — define the environment variable in your terminal before launching VS Code, or add it to your shell profile (.bashrc, .zshrc, $PROFILE on PowerShell). Variables from FROST_VARS override the config file values.
  • Multiple config files — create separate config files per environment (e.g. frost-config-dev.yml, frost-config-prod.yml) and switch between them via the frost.configPath setting in VS Code.
Tip — Per-environment configs in VS Code Set the frost.configPath setting in your workspace .vscode/settings.json: 
{
  "frost.configPath": "frost-config-dev.yml"
}
Each config file can have its own variables: section, making it easy to switch environments without editing values.

Common Patterns

Environment-specific deployments

# frost-config-dev.yml
variables:
  database: DEV_DB
  warehouse: DEV_WH
  admin_role: SYSADMIN

# frost-config-prod.yml
variables:
  database: PROD_DB
  warehouse: PROD_WH
  admin_role: ACCOUNTADMIN
# Deploy to dev
frost deploy -c frost-config-dev.yml

# Deploy to prod
frost deploy -c frost-config-prod.yml

CI/CD with variables

# GitHub Actions — inject variables from secrets
- run: frost deploy
  env:
    FROST_VARS: '{"database": "${{ secrets.SF_DATABASE }}", "admin_role": "${{ secrets.SF_ROLE }}"}'

Dynamic grants

-- objects/grants/role_grants.sql
GRANT ROLE {{read_role}} TO ROLE {{admin_role}};
GRANT USAGE ON WAREHOUSE {{warehouse}} TO ROLE {{read_role}};

⌨️ Commands

frost provides six commands to manage your Snowflake environment.

Global Flags

init   Scaffold a Project

frost init [directory]

Creates the standard project structure with config, sample SQL, sample CSV, YAML sidecar, .env.example, and .gitignore. If directory is omitted, scaffolds in the current directory. Existing files are never overwritten.

plan   Preview Execution Order

frost plan

Parses all SQL files in the objects folder, resolves inter-object dependencies, and displays the topologically-sorted execution order. No changes are made to Snowflake.

deploy   Deploy to Snowflake

frost deploy [--dry-run] [--no-cortex] [--cortex-model MODEL]
FlagDescription
--dry-runShow what would be deployed without executing
--no-cortexDisable Cortex AI fix suggestions on errors
--cortex-modelCortex LLM model (default: mistral-large2)

Deploys SQL objects in dependency order. Tracks successful deployments via checksums — unchanged objects are skipped on subsequent runs. On failure, an ANSI-rich error report is displayed, optionally with Cortex AI–generated fix suggestions.

load   Load CSV Data

frost load [--dry-run] [--data-folder, -d PATH] [--data-schema SCHEMA]
FlagDescription
--dry-runShow what would be loaded without executing
--data-folder, -dOverride data folder path (default: data/)
--data-schemaTarget schema (default: PUBLIC)

Scans the data folder for CSV files, reads optional YAML sidecars for column type overrides, and loads data into Snowflake. Unchanged files (by checksum) are skipped.

graph   Show Dependency Graph

frost graph

Displays the dependency graph showing how SQL objects reference each other and the resolved deployment order. frost automatically detects what each procedure, function, task, and stream reads from and writes to by analysing SQL definitions — no configuration needed. The graph shows both deployment dependencies and data-flow lineage.

lineage   Interactive Lineage Visual

frost lineage [--output, -o FILE] [--local]
FlagDescription
--output, -oOutput HTML file (default: lineage.html)
--localBuild from local SQL files instead of querying Snowflake

Generates an interactive HTML page with a force-directed graph of your lineage data and opens it in your default browser. By default it queries the FROST.OBJECT_LINEAGE table in Snowflake. Use --local to build from local SQL files without a connection.

test   Run Data Quality Tests

frost test [name] [--data-folder, -d PATH]
Argument / FlagDescription
nameOptional bare file name (no extension) to test a single file
--data-folder, -dOverride data folder (default: data/)

Auto-discovers YAML sidecars in the data folder and runs embedded test definitions against the corresponding CSV files. When name is provided, only tests for that file are executed.

# Run ALL tests in data/
frost test

# Run tests for countries.csv only
frost test countries

# Use a different data folder
frost test -d examples/data

🧪 Data Quality Tests

frost ships with six built-in test types that cover the most common data quality assertions.

1. unique

Asserts that all values in a column are unique (no duplicates).

- name: id_unique
  column: id
  test: unique

2. not_null

Asserts that no values in a column are empty or NULL.

- name: name_not_null
  column: name
  test: not_null

3. accepted_values

Asserts that every value in a column is within an allowed set.

- name: status_values
  column: status
  test: accepted_values
  values: [ACTIVE, INACTIVE, PENDING]

4. row_count

Asserts the file has between min and max rows (both optional).

- name: has_data
  test: row_count
  min: 1

- name: not_too_big
  test: row_count
  max: 10000

5. relationship

Asserts every value in column exists in another CSV file's column (referential integrity).

- name: orders_have_customer
  column: customer_id
  test: relationship
  to: customers
  to_column: id
Note The to field can omit the .csv extension — frost appends it automatically.

6. expression

Evaluates a Python expression per row. The expression has access to all column values as local variables, plus safe builtins: int, float, str, bool, len, abs, min, max, round, sum, sorted.

- name: code_length
  test: expression
  expression: "len(code) >= 2"

- name: price_positive
  test: expression
  expression: "float(price) > 0"

Test Results

frost outputs a colour-coded report for each test:

  PASS  country_code_unique      All 7 values in 'code' are unique
  PASS  country_code_not_null    All 7 values in 'code' are non-null
  FAIL  country_region_valid     2 value(s) in 'region' outside accepted set

📄 YAML Sidecar Format

Each CSV file can have a YAML sidecar (.yml / .yaml) that provides column types and data tests.

# countries.yml — lives next to countries.csv
columns:
  code: VARCHAR(10)
  name: VARCHAR(100)
  region: VARCHAR(50)

tests:
  - name: country_code_unique
    column: code
    test: unique
    description: Every country code must be unique

  - name: country_code_not_null
    column: code
    test: not_null

  - name: country_region_valid
    column: region
    test: accepted_values
    values:
      - North America
      - South America
      - Europe
      - Asia
      - Africa
      - Oceania
      - Antarctica

  - name: countries_not_empty
    test: row_count
    min: 1

  - name: country_code_length
    test: expression
    expression: "len(code) >= 2"

Rules

  • The YAML file name determines which CSV it applies to: countries.yml → countries.csv.
  • The source field in tests is optional; when omitted it defaults to the YAML file's stem + .csv.
  • Both source and to auto-append .csv if no extension is present.
  • The columns: section is used by frost load for type overrides; columns not listed default to VARCHAR.
  • The tests: section is used by frost test.
  • File base-names must be unique within the data folder (e.g., you cannot have users.csv and users.tsv).

🔀 Object Lineage

frost automatically detects what each object reads from and writes to — zero configuration needed.

frost parses procedure, function, task, and stream definitions to auto-detect source/target relationships, giving you a complete data-flow map visible in frost graph and queryable from the FROST.OBJECT_LINEAGE table.

Procedures & Functions

frost reads the LANGUAGE property from the CREATE PROCEDURE header (defaults to SQL when omitted) and applies language-specific analysis strategies. Three languages are supported:

LanguageDetection StrategySources (reads)Targets (writes)
SQL Scans DML/query patterns in the procedure body FROM, JOIN, MERGE … USING INSERT INTO, UPDATE … SET, DELETE FROM, MERGE INTO, COPY INTO, CREATE TABLE … AS, TRUNCATE
JavaScript Extracts SQL strings from snowflake.execute({sqlText: "…"}) and snowflake.createStatement({sqlText: "…"}), then applies SQL patterns to each fragment Same SQL patterns applied per fragment Same SQL patterns applied per fragment
Python Detects Snowpark API calls: session.table(), session.sql(), .save_as_table(), .write_pandas() session.table("…"), SQL patterns inside session.sql("…") .save_as_table("…"), .write_pandas(…, "…")
Note JAVA and SCALA procedures are not currently supported for auto-detection. frost logs an informational message and skips them — use a YAML sidecar to declare lineage for these languages.

SQL Example

Just write your procedures normally — frost maps the data flow automatically:

CREATE OR REPLACE PROCEDURE PUBLIC.REFRESH_SUMMARY()
RETURNS VARCHAR LANGUAGE SQL AS
$$
  INSERT INTO PUBLIC.CUSTOMER_SUMMARY
  SELECT c.id, c.name, SUM(o.amount)
  FROM PUBLIC.ORDERS o
  JOIN PUBLIC.CUSTOMERS c ON o.customer_id = c.id
  GROUP BY c.id, c.name;
$$;

frost detects: reads from PUBLIC.ORDERS + PUBLIC.CUSTOMERS, writes to PUBLIC.CUSTOMER_SUMMARY.

JavaScript Example

CREATE OR REPLACE PROCEDURE PUBLIC.SYNC_DATA()
RETURNS VARCHAR LANGUAGE JAVASCRIPT AS
$$
  snowflake.execute({sqlText: "INSERT INTO PUBLIC.TARGET SELECT * FROM PUBLIC.SOURCE"});
$$;

frost extracts the sqlText string and detects: reads from PUBLIC.SOURCE, writes to PUBLIC.TARGET.

Python (Snowpark) Example

CREATE OR REPLACE PROCEDURE PUBLIC.TRANSFORM_DATA()
RETURNS VARCHAR LANGUAGE PYTHON
RUNTIME_VERSION = '3.8'
PACKAGES = ('snowflake-snowpark-python')
HANDLER = 'run'
AS
$$
def run(session):
    df = session.table("PUBLIC.RAW_DATA")
    result = df.group_by("category").agg({"amount": "sum"})
    result.write.save_as_table("PUBLIC.AGGREGATED_DATA")
$$;

frost detects: reads from PUBLIC.RAW_DATA, writes to PUBLIC.AGGREGATED_DATA. Python import statements are correctly ignored.

Tasks

Snowflake tasks schedule SQL execution on a cron or interval. frost extracts lineage from three aspects of a task definition:

  • AFTER parent_task — the parent task in a task chain is detected as a source (dependency)
  • SYSTEM$STREAM_HAS_DATA('stream') — streams referenced in WHEN clauses are detected as sources
  • Task body (SQL after AS) — scanned for the same DML/query patterns as procedures: FROM, JOIN, INSERT INTO, MERGE INTO, etc.

Task Example

CREATE OR ALTER TASK PUBLIC.TASK_CDC
  WAREHOUSE = 'WH_ETL'
  SCHEDULE = '1 MINUTE'
  WHEN SYSTEM$STREAM_HAS_DATA('PUBLIC.ORDER_CHANGES')
AS
MERGE INTO PUBLIC.DIM_ORDERS t
USING PUBLIC.ORDER_CHANGES s ON t.ID = s.ID
WHEN MATCHED THEN UPDATE SET t.AMOUNT = s.AMOUNT
WHEN NOT MATCHED THEN INSERT (ID, AMOUNT) VALUES (s.ID, s.AMOUNT);

frost detects: reads from PUBLIC.ORDER_CHANGES (stream trigger + USING), writes to PUBLIC.DIM_ORDERS.

Task Chain Example

CREATE OR ALTER TASK PUBLIC.TASK_AGGREGATE
  WAREHOUSE = 'WH_ETL'
  AFTER PUBLIC.TASK_CDC
AS
INSERT INTO PUBLIC.DAILY_SUMMARY
SELECT region, SUM(amount) FROM PUBLIC.DIM_ORDERS GROUP BY region;

frost detects: depends on PUBLIC.TASK_CDC (parent), reads from PUBLIC.DIM_ORDERS, writes to PUBLIC.DAILY_SUMMARY.

Streams

Snowflake streams track row-level changes on a table or view. frost detects the tracked source object from the ON TABLE / ON VIEW / ON EXTERNAL TABLE clause:

Stream Example

CREATE OR ALTER STREAM PUBLIC.ORDER_CHANGES
  ON TABLE PUBLIC.RAW_ORDERS;

frost detects: PUBLIC.ORDER_CHANGES reads from PUBLIC.RAW_ORDERS.

Tip — Complete CDC pipeline lineage When tasks reference streams via SYSTEM$STREAM_HAS_DATA('stream'), frost connects the full chain: TABLE → STREAM → TASK → TARGET TABLE. This gives you end-to-end visibility of CDC (Change Data Capture) pipelines.

Dynamic SQL Detection

frost automatically detects when a procedure body contains dynamic SQL and skips auto-detection entirely, because regex-based pattern matching produces unreliable results on dynamically-constructed queries. The following markers trigger this behaviour:

  • EXECUTE IMMEDIATE
  • IDENTIFIER()
  • SYSTEM$QUERY_REFERENCE
  • RESULTSET cursors
  • String concatenation that builds SQL ('SELECT * FROM ' || var)

When dynamic SQL is detected, frost logs an informational message and produces no lineage for that procedure. Use a YAML sidecar to declare lineage manually for these cases.

YAML Overrides (Optional)

For procedures that frost skips (dynamic SQL) or where auto-detection misses references, place a .yml sidecar next to the SQL file to declare lineage manually:

objects/procedures/
    refresh_summary.sql
    refresh_summary.yml   ← optional override
# refresh_summary.yml — override auto-detected lineage
# When sources or targets are declared, they REPLACE the
# auto-detected ones for that field.
sources:
  - PUBLIC.ORDERS
  - PUBLIC.CUSTOMERS
  - PUBLIC.EXCHANGE_RATES   # accessed via dynamic SQL

targets:
  - PUBLIC.CUSTOMER_SUMMARY

description: >
  Aggregates order data per customer into the summary table.
FieldRequiredDescription
sourcesNoReplaces auto-detected sources when declared
targetsNoReplaces auto-detected targets when declared
descriptionNoHuman-readable description added to lineage output
Tip You only need a YAML sidecar when auto-detection misses references (e.g., tables accessed via EXECUTE IMMEDIATE or IDENTIFIER()) or for unsupported languages (JAVA, SCALA). For most SQL, JavaScript, and Python procedures, auto-detection is sufficient.

Graph Storage

Every time frost deploy runs, the full dependency and lineage graph is persisted to FROST.OBJECT_LINEAGE in Snowflake. This table is truncated and reloaded on each deploy to always reflect the latest state.

ColumnTypeDescription
object_fqnVARCHAR(500)The object that owns this edge
object_typeVARCHAR(100)Object type (TABLE, VIEW, PROCEDURE, TASK, STREAM, …)
edge_typeVARCHAR(20)DEPENDENCY, READS, or WRITES
related_fqnVARCHAR(500)The related object
file_pathVARCHAR(1000)Source SQL file path
descriptionVARCHAR(5000)Optional description
recorded_atTIMESTAMP_NTZWhen the edge was recorded
recorded_byVARCHAR(200)Snowflake user who ran the deploy

Query Examples

-- What does REFRESH_SUMMARY read from?
SELECT related_fqn
FROM FROST.OBJECT_LINEAGE
WHERE object_fqn = 'PUBLIC.REFRESH_SUMMARY'
  AND edge_type = 'READS';

-- What writes to CUSTOMER_SUMMARY?
SELECT object_fqn
FROM FROST.OBJECT_LINEAGE
WHERE related_fqn = 'PUBLIC.CUSTOMER_SUMMARY'
  AND edge_type = 'WRITES';

-- Full lineage graph
SELECT object_fqn, edge_type, related_fqn, description
FROM FROST.OBJECT_LINEAGE
ORDER BY object_fqn, edge_type;

📊 Streamlit Deployment

frost integrates with the Snowflake CLI (snow) to discover, deploy, and manage Streamlit apps directly from the CLI and the VS Code extension.

Prerequisites

  • Snowflake CLI — Install via pip install snowflake-cli-labs or brew install snowflake-cli
  • snowflake.yml — Project definition file in or below your frost project root
  • Connection — A configured Snowflake CLI connection (run snow connection add)

Project Definition — snowflake.yml

frost supports both V1 and V2 Snowflake project definition formats:

V1 Format

# snowflake.yml (V1)
streamlit:
  name: my_dashboard
  main_file: streamlit_app.py
  stage: "@my_stage"
  schema: PUBLIC
  warehouse: COMPUTE_WH
  query_warehouse: QUERY_WH
  title: My Dashboard

V2 Format (entities)

# snowflake.yml (V2)
definition_version: 2
entities:
  sales_report:
    type: streamlit
    main_file: report.py
    stage: "@sales_stage"
    schema: SALES
    warehouse: REPORT_WH
    title: Sales Report
  data_explorer:
    type: streamlit
    main_file: explorer.py
    schema: PUBLIC

CLI Commands

CommandDescription
frost streamlit listShow all discovered Streamlit apps
frost streamlit list --jsonJSON output (for tooling)

VS Code Extension — Streamlit Panel

The Frost sidebar includes a Streamlit panel that provides:

  • Auto-discovery — Scans for snowflake.yml files and lists all Streamlit app definitions
  • Snow CLI status — Shows whether the snow CLI is installed and its path
  • App details — Expand each app to see main file, schema, stage, warehouse, and config file
  • Deploy — Right-click an app or use the toolbar button to deploy via snow streamlit deploy
  • Open URL — Open a deployed app's Snowflake URL in your browser
  • Teardown — Right-click → Teardown to drop an app from Snowflake (with confirmation)
  • File watcher — Automatically refreshes when snowflake.yml changes
Tip — Under the hood frost delegates all deployment to the official Snowflake CLI (snow streamlit deploy). This ensures full compatibility with Snowflake's deployment process, including file staging (PUT), the CREATE STREAMLIT statement, and multi-region support. Simply install snow, configure a connection, and frost handles the rest.

How It Works

  1. Discovery — frost walks the project tree looking for snowflake.yml / snowflake.yaml files
  2. Parsing — Each file is parsed for Streamlit definitions (V1 or V2 format)
  3. Deployment — frost invokes snow streamlit deploy in the app's directory, passing --replace and any configured connection options
  4. Result — Success/failure status and the app URL are displayed in the terminal or VS Code panel

Common frost workflows.

Deploy with variables

# Override variables via CLI
frost deploy --vars '{"admin_role": "ACCOUNTADMIN", "db": "PROD_DB"}'

# Or via environment
export FROST_VARS='{"admin_role": "ACCOUNTADMIN"}'
frost deploy

CI/CD Pipeline

# .github/workflows/frost.yml
name: frost deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install frost-ddl
      - run: frost plan
      - run: frost test -d data/
      - run: frost deploy
        env:
          SNOWFLAKE_ACCOUNT: ${{ secrets.SF_ACCOUNT }}
          SNOWFLAKE_USER: ${{ secrets.SF_USER }}
          SNOWFLAKE_DATABASE: ${{ secrets.SF_DATABASE }}
          SNOWFLAKE_PRIVATE_KEY_PATH: ${{ secrets.SF_KEY_PATH }}
          SNOWFLAKE_PRIVATE_KEY_PASSPHRASE: ${{ secrets.SF_KEY_PASS }}

Relationship Test (referential integrity)

# orders.yml
columns:
  order_id: NUMBER
  customer_id: NUMBER
  amount: FLOAT

tests:
  - name: orders_have_customer
    column: customer_id
    test: relationship
    to: customers        # references customers.csv
    to_column: id

  - name: positive_amounts
    test: expression
    expression: "float(amount) > 0"
